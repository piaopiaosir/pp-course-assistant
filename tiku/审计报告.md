## 1. 严重问题（Critical）

### C-2：`/internal/code` 端点泄露全部源码和 `.env` 文件

- **文件**：`routes.js` L163-L220
- **类型**：安全漏洞

**问题描述**：`/internal/code` 端点在验证 `X-Internal-Key` 后，将全部源码和 `.env` 文件内容作为 JSON 返回（含数据库密码、所有 API 密钥）。默认密钥为弱密码 `internal-secret-key-2024`。

**修复建议**：移除 `.env` 文件返回；移除默认密钥，强制配置 `INTERNAL_API_KEY`；增加 IP 白名单和频率限制。

## 2. 高级别问题（High）

### H-8：`checkTokenStatus` 中 DECIMAL 与整数比较导致误封号

- **文件**：`auth.js` L280
- **类型**：数据类型 / 功能缺陷

**问题描述**：`remaining_count < 1` 的判断对 DECIMAL(10,1) 类型不合理：余额 0.5~0.9 次的用户被自动拉黑，丢失余额。

**修复建议**：改为 `remaining_count < 0.1`（最小可扣单位）。


---

### H-12：AI 模式余额检查不预留次数

- **文件**：`ai-mode.js` L424-L433
- **类型**：边界条件

**问题描述**：调用前仅 `checkOnly=true` 检查余额 ≥1，不预留。AI 调用期间（可能耗时数十秒），其他请求可能消耗掉余额，导致实际扣减失败。

**修复建议**：采用"先扣后返"策略：先扣 1 次预留，AI 失败时返还。

---

### H-13：`/welfare` 端点无条件解除黑名单

- **文件**：`routes.js` `/welfare` 路由 L589-L591
- **类型**：功能缺陷

**问题描述**：福利领取时自动 `is_blacklisted = 0`，不区分黑名单原因。若 token 因滥用被封（而非次数耗尽），此操作会不当解封。

**修复建议**：仅在 `remaining_count < 1` 且非永久封禁时解封；或增加黑名单原因字段。

---

### H-15：`fetchAISupplement` 深度思考默认用同一模型

- **文件**：`normal-mode.js` L26-L28
- **类型**：业务逻辑

**问题描述**：`AI_MODEL_NORMAL` 和 `AI_MODEL_NORMAL_THINKING` 都默认 `deepseek-v4-flash`，"深度思考"仅启用 thinking 参数，未切换更强模型。可能得出相似错误答案。

**修复建议**：默认 `AI_MODEL_NORMAL_THINKING` 为 `deepseek-v4-pro`。

---

## 3. 中等级别问题（Medium）

### M-7：`recheck.js` 中 AbortController 错误处理丢失类型信息

- **文件**：`recheck.js` L113-L139
- **类型**：错误处理

**问题描述**：超时 `AbortError` 被包装为通用的 `new Error("请求错误: ...")`，调用方无法区分超时和网络错误。

---


### M-12：DB 迁移代码脆弱，部分失败后继续运行

- **文件**：`config.js` 多个迁移 `try/catch`
- **类型**：可靠性

**问题描述**：多个迁移步骤各自 `try/catch`，某步失败只打日志继续执行。可能导致 DB 结构不完整。迁移应记录执行状态，失败的步骤在下次启动时重试。

---

### M-13：过度 `console.log` 包含敏感信息

- **多文件**：`routes.js`、`auth.js`、`tiku.js`、各 mode 文件
- **类型**：安全

**问题描述**：日志输出 token、IP、API key 前缀等敏感信息。应引入 logger 模块（如 pino），敏感信息脱敏，可配置日志级别。

---

### M-17：`DEFAULT_GLOBAL_STATS` 包含 DB 中不存在的历史字段

- **文件**：`config.js`
- **类型**：代码质量

**问题描述**：`hunyuan_calls_count`、`deepseek_thinking_calls_count` 等字段在 CREATE TABLE SQL 中不存在，`{ ...DEFAULT_GLOBAL_STATS, ...stats }` 合并时永远从默认值取，无实际意义。应清理。

---

### M-19：`/admin/delete` 无操作审计日志

- **文件**：`admin-routes.js`
- **类型**：安全审计

**问题描述**：管理员删除操作只验证会话后执行 SQL DELETE，不写入 `admin_access_logs`。应记录审计日志。

---

### M-21：校验模式 `checkOnly=true` 时未检查第二步余额

- **文件**：`verify-mode.js` L848-L862
- **类型**：边界条件

**问题描述**：第一步扣除费用后未检查余额是否足以支付第二步深度思考。应预估第二步最大消耗并提前检查。

---

### M-23：正常模式缓存命中和 AI 补充扣费策略不一致

- **文件**：`normal-mode.js`
- **类型**：业务逻辑矛盾

**问题描述**：缓存命中扣 0.8 次，正常模式 AI 扣 1 次，AI 模式按 token 扣 1-10+ 次。正常模式 AI 可能消耗大量 token（深度思考重试）但只扣 1 次，与 AI 模式的按量计费矛盾。应统一扣费策略。

---

### M-29：`config.js` 数据库建表 COMMENT 中存在乱码字符

- **文件**：`config.js` L314 等多处
- **类型**：编码

**问题描述**：数据库建表语句的 COMMENT 部分出现 `'���同上报累计次数'` 等乱码，可能是文件编码转换问题。不影响运行但影响可读性。

**修复建议**：确认文件编码为 UTF-8，重新输入缺失的中文字符。

## 4. 低级别问题（Low）


### L-11：`calculatePollInterval` 永远返回固定值 1000

- **文件**：`query-tasks.js`
- **类型**：死代码

该端点暴露的值永远固定，无动态计算逻辑。若无动态需求可改为常量。

---

### L-13：响应 DTO 结构不一致

- **多文件**：各端点返回的 `data` 结构不统一
- **类型**：代码规范

有的含 `source`，有的含 `sponsorUrl`，有的含 `limitedMode`。建议定义统一的 Response DTO。

---

### L-15：小数扣减导致余额碎片

- **文件**：`normal-mode.js` L319
- **类型**：业务逻辑

缓存扣 0.8、AI 扣 1 次等小数操作长期累积后会出现 0.2、0.4 等无法使用的碎片余额。建议统一为整数扣减或改用 DECIMAL(10,2)。

---

## 5. 架构级建议

### A-2：引入请求级日志替代 `console.log`

使用 pino 或类似库，支持：
- 请求级 context（`requestId` 自动注入）
- 日志分级（debug/info/warn/error）
- 敏感字段自动脱敏

### A-5：密钥管理外置

所有 API 密钥（Tavily 50 个、题库海、DeepSeek、302.AI 等）移至 `.env` 或 DB 配置表，支持运行时热更新。`/internal/code` 端点必须移除 `.env` 泄露。
