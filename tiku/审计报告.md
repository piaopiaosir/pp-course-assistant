## 一、严重问题（Bug / 逻辑错误）

### 1.1 `tavily-search.js` - `switchToNextAvailableKey` 索引越界风险

- **文件**: `tavily-search.js`
- **位置**: 第 70 行
- **问题**: 循环 `for (let i = currentKeyIndex; i < TAVILY_KEY_COUNT; i++)` 中，`currentKeyIndex` 是 1-based 索引，但 `TAVILY_KEYS` 数组是 0-based。当 `currentKeyIndex = TAVILY_KEY_COUNT` 时，`TAVILY_KEYS[i]` 会越界返回 `undefined`。
- **影响**: 当最后一个密钥失效后尝试切换时，可能访问不存在的数组元素。
- **建议**: 将循环改为 `for (let i = currentKeyIndex - 1; i < TAVILY_KEY_COUNT; i++)`，统一使用 0-based 索引。

### 1.2 `recheck.js` - `stats` 变量与函数参数名冲突

- **文件**: `recheck.js`
- **位置**: 第 41 行（模块级 `stats` 对象）与第 58 行（`callTikuApi` 函数内的局部 `stats` 变量）
- **问题**: 模块顶层定义了 `let stats = { ... }`（第 41 行），但在 `callTikuApi` 函数中（第 58 行）又用 `const stats = await db.prepare(...)` 声明了同名局部变量。虽然 JavaScript 作用域规则允许这样做（局部变量遮蔽外层），但极易造成混淆和维护错误。
- **影响**: 代码可读性降低，后续维护者可能误操作外层 `stats` 对象。
- **建议**: 将 `callTikuApi` 中的局部变量重命名为 `globalStats` 或 `tikuStats`。

## 二、中等问题（逻辑矛盾 / 设计缺陷）

### 2.1 `verify-mode.js` - 校验模式扣费逻辑复杂且存在重复扣费风险

- **文件**: `modes/verify-mode.js`
- **位置**: 第 458-476 行、第 689-703 行、第 784-798 行、第 892-925 行
- **问题**: 校验模式的扣费逻辑分散在多个位置：
  1. 入口处余额检查（第 467 行，`checkOnly=true` 不实际扣费）
  2. 缓存命中扣 0.8 次（第 511 行）
  3. 题库+AI 一致时扣 1 次基础 + AI token 费用（第 697-698 行）
  4. 不一致时第一轮扣 1 次基础 + AI token 费用（第 792-793 行）
  5. 第二轮深度思考扣第二轮 AI token 费用（第 914 行）
  
  在 `checkOnly=false` 的第二轮请求中（第 892-925 行），代码重新计算了第一轮费用 `step1Cost`（第 900 行），但这个费用在第一次 `checkOnly=true` 请求时已经扣过了。虽然代码注释说"此处仅用于总费用统计"，但 `totalCost = step1Cost + step2Cost`（第 912 行）会导致返回给客户端的 `cost` 字段是第一轮+第二轮的总和，而实际上第一轮费用在第一次请求时已经单独返回过。
- **影响**: 客户端显示的消耗次数可能不准确，用户可能困惑为什么第二轮请求显示的费用包含了第一轮的费用。
- **建议**: 明确区分"本轮实际扣费"和"累计总费用"，在第二轮请求中只返回第二轮的消耗。

### 2.2 `normal-mode.js` - 排序题特殊处理逻辑与正常流程不一致

- **文件**: `modes/normal-mode.js`
- **位置**: 第 339-433 行
- **问题**: 排序题（type=13）直接跳过所有题库查询，使用 AI 补充。但这段代码（第 339-433 行）与正常流程（第 435-622 行）是并列的，意味着排序题不会走缓存查询、题库查询等正常流程。然而，缓存查询在正常流程的最开始（第 271 行），排序题分支在缓存查询之后，导致排序题也无法命中缓存。
- **影响**: 排序题每次都需要调用 AI，即使之前已经查询过并缓存了答案。
- **建议**: 将排序题的缓存检查逻辑提前，或在排序题分支内部也加入缓存查询。

### 2.3 `auth.js` - `decrementCount` 中的 `checkOnly` 模式不验证 `is_blacklisted`

- **文件**: `auth.js`
- **位置**: 第 413-442 行
- **问题**: 当 `checkOnly=true` 时，`decrementCount` 函数只检查 `remaining_count < count`，不检查 `is_blacklisted` 状态。这意味着已被拉黑的 Token 在 `checkOnly` 模式下仍会返回 `success: true`。
- **影响**: 校验模式入口处的余额检查（`verify-mode.js` 第 467 行）使用 `checkOnly=true`，可能导致已被拉黑的 Token 通过余额检查。
- **建议**: 在 `checkOnly` 模式下也检查 `is_blacklisted` 状态。

### 2.4 `tavily-search.js` - `getAvailableTavilyKey` 中 `loadInvalidKeys` 可能未完成就被调用

- **文件**: `tavily-search.js`
- **位置**: 第 31-45 行
- **问题**: `loadInvalidKeys()` 在第 45 行被调用，但没有 `await`。这是一个异步函数，如果数据库连接尚未就绪，加载可能失败。后续的 `getAvailableTavilyKey` 可能在 `loadInvalidKeys` 完成之前就被调用，导致失效密钥列表为空。
- **影响**: 启动后短时间内，已失效的 Tavily 密钥可能被再次使用，造成不必要的 API 调用失败。
- **建议**: 在模块初始化时 `await loadInvalidKeys()`，或使用初始化锁确保加载完成后再处理请求。

### 2.5 `cache.js` - `saveAnswerToCache` 的 `ON DUPLICATE KEY UPDATE` 可能覆盖 `is_correct` 状态

- **文件**: `tiku/cache.js`
- **位置**: 第 106-108 行
- **问题**: `INSERT ... ON DUPLICATE KEY UPDATE` 语句中，`is_correct` 被设置为传入的 `isCorrectValue`（第 107 行）。如果调用方传入 `isCorrect = null`（默认值），会覆盖数据库中已有的 `is_correct` 值（例如之前已被标记为 1 的已验证答案）。
- **影响**: 当缓存已存在且 `is_correct=1`，但新调用 `saveAnswerToCache` 时未传入 `isCorrect` 参数，会导致 `is_correct` 被重置为 NULL。
- **建议**: 在 UPDATE 子句中使用 `COALESCE(VALUES(is_correct), is_correct)` 或条件判断，仅当传入值非 NULL 时才更新。

### 2.6 `api.js` - `fetchAnswer` 中 `incrementTotalQueries` 被调用两次

- **文件**: `tiku/api.js`
- **位置**: 第 407 行和第 459 行
- **问题**: 在 `fetchAnswer` 函数中，当 API 调用成功时，`incrementTotalQueries('tiku')` 在第 407 行被调用；当 API 调用失败但非次数耗尽时，同一函数在第 459 行也被调用。这意味着一次题库海查询可能被计数两次（如果第一次调用失败但非次数耗尽，第二次调用成功）。
- **影响**: 统计数据不准确，总查询次数可能高于实际查询次数。
- **建议**: 确保每次 `fetchAnswer` 调用只增加一次总查询计数。

## 四、流程逻辑设计问题

### 4.1 正常模式查询链：先查缓存再扣费的设计缺陷

- **文件**: `modes/normal-mode.js`
- **位置**: 第 271-320 行
- **问题**: 缓存命中时扣 0.8 次，但如果缓存答案校验失败（第 305-306 行），代码继续走正常查询流程。此时 0.8 次已经被扣除，但后续正常查询还会再扣 1 次，导致用户被扣 1.8 次。
- **影响**: 缓存答案校验失败时用户被多扣费。
- **建议**: 缓存答案校验失败时，应先退还已扣的 0.8 次，或改为在确认缓存答案有效后再扣费。

### 4.2 校验模式两步请求的状态管理依赖客户端配合

- **文件**: `modes/verify-mode.js`
- **位置**: 第 540-589 行、第 780-813 行
- **问题**: 校验模式设计为两步：第一步 `checkOnly=true` 返回 202 状态码，第二步 `checkOnly=false` 执行深度思考。但这两步之间没有服务端状态管理（如 session），完全依赖客户端正确发送第二步请求。如果客户端不发送第二步请求，第一步已扣除的费用不会退还。
- **影响**: 客户端异常或网络问题导致第二步未执行时，用户损失第一步费用。
- **建议**: 增加超时自动退还机制，或使用服务端任务状态追踪。

### 4.3 `lockToken` 预锁定机制的异常释放策略

- **文件**: `auth.js`
- **位置**: 第 630-635 行
- **问题**: `releaseToken` 函数在异常时调用，注释说明"不返还次数，锁定时的扣减保留"。这意味着如果 AI 调用因网络超时等不可控因素失败，用户将损失预锁定的次数（可能高达 20 次）。
- **影响**: 用户可能因网络问题损失大量次数，体验较差。
- **建议**: 区分"用户主动取消"和"系统异常"，对系统异常（如 API 超时、网络错误）应退还预锁定次数。

### 4.4 `routes.js` 中间件中 IP 封禁检查在频率限制之前

- **文件**: `routes.js`
- **位置**: 第 62-70 行（IP 封禁检查）、第 79-92 行（频率限制检查）
- **问题**: 封禁检查在频率限制之前执行。如果一个 IP 已被封禁，它仍然会经过频率限制检查（虽然封禁检查会先拦截）。但封禁 IP 的请求仍会触发 `recordIpViolation`（第 83 行），导致违规计数继续增加。
- **影响**: 已被封禁的 IP 违规计数会持续增长，可能导致封禁时间不断延长。
- **建议**: 封禁检查通过后直接返回，不再执行后续的频率限制和违规记录。

### 4.5 `api.js` - `fetchHiveNet` 中缓存所有题目可能造成大量数据库写入

- **文件**: `tiku/api.js`
- **位置**: 第 831-885 行
- **问题**: Hive-Net 每次查询返回的所有题目（可能多达数十道）都会被缓存到数据库。这些题目可能与当前用户查询的题目无关，造成不必要的数据库写入和存储开销。
- **影响**: 数据库写入压力增大，存储空间浪费。
- **建议**: 考虑只缓存与当前查询相关的题目，或将批量缓存改为异步低优先级任务。

## 五、安全性问题

### 5.1 `routes.js` - 内部代码同步接口暴露完整源代码

- **文件**: `routes.js`
- **位置**: 第 159-226 行（`/internal/code` 路由）
- **问题**: 该接口在有正确 `INTERNAL_API_KEY` 的情况下，返回所有源代码文件和 `.env` 文件内容。如果 `INTERNAL_API_KEY` 泄露，攻击者可以获取完整的服务器代码和环境变量（包括所有 API 密钥）。
- **影响**: 严重安全风险。`.env` 文件包含所有第三方 API 密钥。
- **建议**: 
  1. 不要通过 HTTP 接口暴露 `.env` 文件内容
  2. 对 `INTERNAL_API_KEY` 使用更强的认证机制
  3. 添加 IP 白名单限制

### 5.2 `auth.js` - Token 生成使用 `Math.random()`

- **文件**: `auth.js`
- **位置**: 第 196 行
- **问题**: `generateValidToken` 使用 `Math.random()` 生成 12 位随机种子。`Math.random()` 不是密码学安全的随机数生成器，其种子可被预测。
- **影响**: 攻击者可能预测 Token 生成规律，伪造有效 Token。
- **建议**: 使用 `crypto.randomInt()` 或 `crypto.randomBytes()` 替代。

### 5.3 `ip-security.js` - IP 归属地查询使用 HTTP 明文

- **文件**: `ip-security.js`
- **位置**: 第 99 行
- **问题**: IP 归属地查询使用 `http://ip9.com.cn/get?ip=...`（非 HTTPS），且将用户 IP 通过明文传输。
- **影响**: 中间人攻击可能篡改归属地信息或窃取 IP 数据。
- **建议**: 改用 HTTPS 端点，或使用支持 HTTPS 的 IP 归属地服务。

### 5.4 `admin/session.js` - 管理面板密码硬编码风险

- **文件**: `admin/session.js`（未直接读取但根据项目结构推断）
- **问题**: 管理面板的密码比较使用 `safeComparePassword`，但密码来源通常是环境变量。如果 `.env` 文件被泄露（如问题 5.1），管理面板将完全暴露。
- **建议**: 确保管理面板密码足够复杂，并考虑添加二次认证。

## 六、性能问题

### 6.1 `query-tasks.js` - `recentlyQueriedQuestions` Map 可能无限增长

- **文件**: `query-tasks.js`
- **位置**: 第 30-65 行
- **问题**: `recentlyQueriedQuestions` Map 的最大容量设为 50000，清理定时器每 60 秒运行一次。在高并发场景下，如果每秒新增超过 833 条记录，Map 可能在清理间隔内超过容量限制。
- **影响**: 内存使用可能超出预期。
- **建议**: 考虑使用 LRU 缓存替代普通 Map，或缩短清理间隔。

### 6.2 `ip-security.js` - `ipRequestCache` Map 无上限增长风险

- **文件**: `ip-security.js`
- **位置**: 第 212-272 行
- **问题**: `ipRequestCache` 最大容量设为 10000，清理定时器每 60 秒运行一次。在高并发或 DDoS 攻击下，不同 IP 数量可能迅速超过 10000。
- **影响**: 内存使用激增，可能导致服务不稳定。
- **建议**: 使用更积极的淘汰策略，或使用 Redis 等外部存储。

### 6.3 `tavily-search.js` - 递归调用可能导致深度调用栈

- **文件**: `tavily-search.js`
- **位置**: 第 238-403 行
- **问题**: `tavilySearch` 函数在密钥切换时递归调用自身（第 315、320、331、350 行），最大递归深度 30。虽然深度限制合理，但每次递归都会创建新的 `getAvailableTavilyKey` 数据库查询。
- **影响**: 在最坏情况下，一次搜索可能触发 30 次数据库查询。
- **建议**: 将递归改为循环，避免不必要的数据库查询。

### 6.4 `routes.js` - 每次请求都执行 IP 归属地查询

- **文件**: `ip-security.js` 第 184-209 行（`logIpAccess` 函数）
- **位置**: 新 IP 首次访问时调用 `getIpLocation`
- **问题**: `getIpLocation` 发起外部 HTTP 请求（超时 3 秒），会阻塞请求处理。虽然 `logIpAccess` 在路由中间件中以 `.catch(() => {})` 方式调用，但 `await` 仍然会等待。
- **影响**: 新 IP 首次访问时响应延迟增加最多 3 秒。
- **建议**: 将 IP 归属地查询改为完全异步（不 await），或使用本地 IP 数据库。

## 七、代码质量问题

### 7.1 过度使用 `console.log`

- **涉及文件**: 几乎所有文件
- **问题**: 大量 `console.log` 用于调试和状态输出，没有使用日志级别区分。在生产环境中会产生大量日志，影响性能和可读性。
- **建议**: 引入日志库（如 `winston` 或 `pino`），使用日志级别（debug/info/warn/error）。

### 7.2 魔法数字散布各处

- **涉及文件**: 多个文件
- **示例**:
  - `auth.js` 第 220 行：`40`（新用户初始次数）
  - `routes.js` 第 252 行：`5000`（全局每日配额）
  - `ip-security.js` 第 277 行：`100`（受限模式每日限额）
  - `query-tasks.js` 第 2 行：`5 * 60 * 1000`（任务过期时间）
- **建议**: 将这些值提取到 `config/app-config.js` 中作为命名常量。

### 7.3 错误处理不一致

- **问题**: 部分函数使用 try-catch 并返回错误对象（如 `{ code: 500, msg: ... }`），部分函数直接抛出异常，部分函数只打印日志后静默失败。
- **示例**: 
  - `stats.js` 中的函数全部静默捕获异常
  - `api.js` 中的函数返回结构化错误
  - `helpers.js` 中的 `extractJsonFromContent` 返回 null
- **建议**: 统一错误处理策略，建议全部使用结构化错误返回。

### 7.4 模块间循环依赖风险

- **涉及文件**: `tiku.js` → `tiku/api.js` → `tiku/cache.js` → `tiku/helpers.js` → `utils.js`
- **问题**: 虽然当前设计避免了循环依赖（通过 `tiku.js` 作为聚合入口），但依赖链较长，且 `api.js` 同时依赖 `cache.js` 和 `helpers.js`，而 `cache.js` 又依赖 `helpers.js`。
- **建议**: 考虑使用依赖注入或事件驱动模式减少模块间直接依赖。

### 7.5 数据库连接未统一管理

- **问题**: `pool.getConnection()` 在多处被直接调用（`auth.js`、`routes.js`、`tavily-search.js`），连接释放逻辑分散。如果某处忘记 `conn.release()`，会导致连接泄漏。
